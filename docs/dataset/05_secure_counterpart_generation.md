# 05 Secure Counterpart Generation

## Purpose

This document defines the methodology used to generate secure counterparts for the SecurityEval-derived vulnerable samples.

The goal is to create secure implementations that preserve the original intended functionality while mitigating the target CWE using authoritative MITRE CWE guidance.

---

## Inputs

Each secure counterpart is generated using:

- The original vulnerable implementation
- The sample metadata
- The CWE ID
- The CWE name
- MITRE CWE description
- MITRE CWE mitigation guidance

---

## Source of Security Guidance

Security guidance is derived from MITRE CWE data stored in:

```text
datasets/securityeval_expanded/metadata/cwe_guidance_data.json

Human-readable guidance files are stored in:

datasets/securityeval_expanded/metadata/cwe_guidance/

Secure Rewrite Objective

For each vulnerable sample, the secure counterpart should:

Preserve intended functionality
Remove or mitigate the target CWE
Follow CWE-specific mitigation guidance
Avoid unnecessary rewrites
Avoid introducing unrelated functionality
Keep the code readable
Keep the implementation executable when possible
Generation Rule

Secure counterparts must not be generated using only a generic instruction such as:

Make this code secure.

Instead, secure generation must include:

Vulnerable code
CWE ID
CWE name
Weakness summary
Mitigation guidance
Explicit instruction to preserve behavior while mitigating the target weakness
Output Structure

Secure counterparts are stored in:

datasets/securityeval_expanded/secure/

Each secure sample should eventually follow:

SE-CWE<ID>-<NUMBER>/
├── secure.py
├── metadata.json
└── generation_notes.md
Metadata Requirements

Each secure sample metadata file should include:

sample_id
source_vulnerable_sample_id
task_id
cwe_id
cwe_name
vulnerability_status
source
generation_method
secure_generation_prompt_version
context_profile_version
validation_status
notes
Validation Requirements

Secure counterparts should be validated through:

Syntax review
Manual security review
Comparison against the vulnerable version
CWE mitigation alignment
Bandit analysis where applicable
CodeQL analysis where applicable
Reproducibility Requirements

The generation process should record:

Model used
Prompt version
CWE guidance source
Input vulnerable sample
Output secure sample
Manual review status
Tool-validation status
Methodology Statement

Secure counterparts are generated as CWE-guided secure rewrites of SecurityEval vulnerable reference implementations.

The secure examples are not treated as automatically correct simply because they were generated by an LLM.

Each secure counterpart must be reviewed and validated before being marked as part of the final benchmark.
